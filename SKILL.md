# Spec-First Methodology

Spec-First — это принцип, при котором ни одна строка кода не пишется, пока проект полностью не описан в документации. Спецификация — это ТЗ для AI-агентов, а не для людей. Чем конкретнее описаны таблицы, эндпоинты, компоненты — тем автономнее работает Claude Code.

## Когда применять

Используй этот скилл, когда пользователь:
- Начинает новый проект и хочет использовать Claude Code
- Хочет, чтобы Claude Code работал более автономно
- Спрашивает, как структурировать проект для AI-разработки
- Хочет написать CLAUDE.md, субагентов, rules или skills
- Упоминает "spec-first", PROJECT_IDEA.md или техническую спецификацию для Claude Code

## Пайплайн: 6 слоёв (с нулевым для работы с заказчиком)

Каждый слой — документ, который трансформирует данные предыдущего в более конкретный формат. Слой 0 используется только при работе со сторонними заказчиками.

---

### Слой 0 — Клиентский бриф (CLIENT_BRIEF.md)

Используй этот слой, когда пользователь работает с внешним заказчиком и нужно собрать информацию до начала проекта.

**Принцип работы:**
1. Пользователь описывает проект заказчика — хотя бы в одной фразе
2. Claude анализирует домен и тип продукта — генерирует кастомный бриф с релевантными вопросами
3. Заказчик заполняет бриф в свободном темпе
4. Заполненный бриф возвращается в Claude — он автоматически собирает PROJECT_IDEA.md без уточнений

**Ключевое правило:** бриф пишется на языке заказчика — без технического жаргона.

**Домены и сигнальные слова:**

| Домен | Сигналы | Добавить блоки |
|---|---|---|
| Сервисный бизнес | салон красоты, клиника, студия, запись, мастер | расписание, запись клиентов, лояльность |
| SaaS / B2B | отчёты, аналитика, автоматизация, Google Sheets | интеграции, роли пользователей, экспорт |
| E-commerce | продажи, товары, каталог, заказы, доставка | каталог, корзина, склад, логистика |
| Маркетплейс | продавцы и покупатели, объявления, комиссия | стороны платформы, доверие, споры |
| Внутренний инструмент | сотрудники, отдел, CRM, согласование | оргструктура, доступы, системы |
| Бот / автоматизация | бот, уведомления, Telegram, рассылки | триггеры, каналы доставки, расписание |
| Мобильное приложение | iOS, Android, приложение | платформа, офлайн-режим, push |

Примеры готовых брифов по доменам → references/templates.md

---

### Слой 1 — Документ идеи (PROJECT_IDEA.md)

**10 обязательных разделов:**
1. Проблема — конкретная боль с числами
2. Решение — пошаговый процесс: каждый шаг станет модулем
3. Почему сейчас — тайминг рынка
4. Целевая аудитория — основная и вторичная
5. Архитектура — слои системы с диаграммой, стек с обоснованием
6. Монетизация — планы, цены, гейтинг фич
7. Конкуренты — таблица
8. План запуска — фазы с целями и метриками
9. Риски — таблица: риск, вероятность, митигация
10. Техдетали — структура репозитория, ключевые таблицы

---

### Слой 2 — Техническая спецификация

**6 обязательных блоков каждого модуля:**
1. User Stories
2. Модель данных — таблицы, типы (uuid, text, jsonb, timestamptz), RLS
3. API и серверная логика — эндпоинты, методы, коды ошибок
4. Экраны и компоненты — состояния: loading, error, empty, success
5. Бизнес-логика — правила валидации, ограничения, формулы
6. Крайние случаи — нет данных, API не отвечает

---

### Слой 3 — Генератор конфигурации

**Что генерируется:**
- CLAUDE.md (до 120 строк) — главный конфиг
- .claude/agents/*.md (4–6 файлов) — субагенты
- .claude/rules/*.md (3–5 файлов) — контекстные правила
- .claude/skills/*.md (2–3 файла) — навыки
- SPEC_TEMPLATE.md — шаблон для новых фич
- MCP-команды — Context7, Supabase, GitHub

---

### Слой 4 — Субагенты

| Субагент | Модель | Ответственность |
|---|---|---|
| database-architect | opus | Схема БД, миграции, RLS, индексы |
| backend-engineer | opus | API, Server Actions, Edge Functions |
| frontend-developer | sonnet | UI, компоненты, формы, навигация |
| qa-reviewer | sonnet | Code review, тесты RLS, безопасность |
| payments-specialist | opus | Stripe, подписки, вебхуки, биллинг |
| ai-agent-architect | opus | Продакшн AI-агенты для пользователей |

**Важно:** qa-reviewer получает только Read, Bash, Glob, Grep — без Write и Edit.

---

### Слой 5 — Промпт-инструкция

Три типа управляющих документов:
1. Идея → Спецификация
2. Спецификация → Конфигурация
3. Конфигурация → Сборка

---

## Как работать с этим скиллом

0. Если проект от внешнего заказчика → создай бриф (Слой 0)
1. Если у пользователя только идея → помоги написать PROJECT_IDEA.md (Слой 1)
2. Если есть PROJECT_IDEA.md → генерируй техническую спецификацию (Слой 2)
3. Если есть спецификация → генерируй конфигурацию: CLAUDE.md, субагенты, rules, skills (Слой 3–4)
4. Если пользователь просит промпт-инструкцию → создай управляющий документ (Слой 5)
5. Если пользователь готов к сборке → проведи по чеклисту из references/checklist.md

## Антипаттерны

- Код без спецификации — через день полная переделка
- Абстрактная спецификация — нужны конкретные таблицы, поля, эндпоинты
- Раздутый CLAUDE.md — максимум 120 строк
- Все субагенты на Opus — Sonnet в 2× дешевле для UI и code review
- Ревьюер с правами на запись — qa-reviewer только Read, Bash, Glob, Grep
- Без Context7 MCP — Claude пишет по устаревшим API
